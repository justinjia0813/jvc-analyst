# vc-analyst 工作流说明（v0）

> 状态：初始草稿。目的是把"早期投资经理一天到一周内会做的事"拆成可被 Claude 协助的、有标准产出物的环节。
> 适用范围：Pre-seed ~ Series B、人民币基金、中国市场。
> 这份文件是约定，不是软件。每个"环节"对应未来一个 skill / slash command（待实现）。

---

## 0. 设计原则

1. **环节独立、产出物明确**：每个环节有一个固定的 Markdown 模板和归档位置。
2. **判断不替代人**：工具只整理证据、提问、找反面，**不替你做投/不投决策**。
3. **可中断、可续接**：项目档案是文件系统驱动，任何时候新对话都能根据档案恢复上下文。
4. **本地优先**：所有素材和分析留在本地仓库，不外发。

## 1. 已有可复用 skills（外部）

下列 skill 已独立存在，本工作流**直接复用**而非重新实现。每个 skill 的接入位置见后文环节描述。

本仓库已在 `skills/` 下收录这些外部 skill 的本地入口说明，并在 `library/skill-registry.md` 维护统一索引。外部仓库仍是实现代码的事实来源，本仓库只记录它们如何接入 VC 工作流。

### 1.1 `/meeting-notes` — 访谈纪要生成器
- 仓库：https://github.com/justinjia0813/meeting-notes
- 输入：转写文本 + 用户笔记（JSON 结构）
- 输出：6 段式 .docx 文件（公司概览 / 核心技术 / 团队 / 产品 / 商业化 / 融资），文件名格式 `{date}_{project}_访谈纪要.docx`
- 依赖：`python-docx`
- **接入位置**：环节 A `/intake`、环节 E `/founder-sync`、环节 G `/ref-check`
- **关键搭档**：`/asr`——先用 `/asr` 把音频/视频转写成文本，再交给 `/meeting-notes` 结构化
- 注：6 段模板恰好和早期投资项目登记需要的信息维度一致，所以这个 skill 在本工作流里承担了"结构化纪要"的核心职责

### 1.2 `/invoice-manager` — 发票整理器（运营辅助）
- 仓库：https://github.com/justinjia0813/invoice-manager
- 输入：PDF 发票（增值税 / 电子 / 酒店）→ OCR 识别
- 输出：报销汇总 Excel + 按差旅归档的重命名 PDF（`archive/{月份}{地点}{项目}差旅/`）
- 依赖：poppler、tesseract、tesseract-lang（系统）+ `openpyxl pdf2image pytesseract`（Python）
- **接入位置**：不在投资决策流程内，属于**运营辅助**——VC 经理频繁出差尽调，差旅报销是高频低价值动作，由此 skill 自动化
- 调用时机：每次出差结束后、或月底报销周期

### 1.3 `/asr` — 语音转写（搭档 skill）
- 用途：把访谈/会议的音频或视频转成文本
- **接入位置**：所有涉及访谈的环节（E `/founder-sync`、G `/ref-check`）的**前置步骤**
- 标准链路：`/asr` → 得到转写文本 → `/meeting-notes` → 得到结构化 .docx

## 2. 项目档案目录约定

每个被正式跟踪的项目，在 `projects/{company-slug}/` 下展开：

```
projects/{company-slug}/
├── 00-source/              # 原始素材：deck、财务表、访谈录音转写、官网抓取
├── 01-prescreen.md         # 初筛纪要（去/留判断 + 理由）
├── 02-dd-team.md           # 团队尽调
├── 02-dd-product.md        # 产品 / 技术尽调
├── 02-dd-market.md         # 市场 / 竞争尽调（可链接到 tracks/）
├── 02-dd-finance.md        # 财务 / 单位经济模型尽调
├── 03-founder-sync.md      # 与创始人的对话准备和纪要
├── 04-bear-case.md         # 反向论证：找最锋利的不投理由
├── 05-ref-check.md         # 客户、前同事、上下游背调汇总
├── 06-ic-memo.md           # 投决会备忘录
└── 99-decision.md          # 最终决策、估值条款摘要、复盘埋点
```

赛道研究放 `tracks/{track-slug}/`，可复用模板和数据库放 `library/`。

## 3. 工作流环节（待实现的 skill 列表）

每个环节后续会落成一个独立的 skill。环节之间是**默认顺序但允许跳步**。

### 环节 A：`/intake` 项目登记
- **输入**：BP / deck / 一句话描述；如果是首次接触会的录音/视频，先走 `/asr` → `/meeting-notes`
- **动作**：建 `projects/{slug}/` 骨架，提取公司基础信息到 `intake.md`
- **产出**：项目档案目录 + 一页基础事实卡（成立时间、轮次、行业、核心人、上一轮投资人）
- **复用 skill**：当输入是首次会议录音时，`/meeting-notes` 生成的 6 段 .docx 直接归入 `00-source/`，并提取摘要进 `intake.md`

### 环节 B：`/prescreen` 初筛
- **输入**：`intake.md` + deck
- **动作**：用 5–7 个早期投资关心的核心问题问自己（市场容量、痛点真实性、为什么是这个团队、当下时机、显性风险）
- **产出**：`01-prescreen.md`，明确"继续深尽调 / 放弃 / 观察 N 个月"
- **硬约束**：必须显式写出 bear case 的雏形，不允许只列优点

### 环节 C：`/track-research` 赛道研究
- **输入**：赛道名称
- **动作**：行业图谱 → 关键玩家 → 价值链 → 监管 → 增长驱动 → 风险因素
- **产出**：`tracks/{slug}/landscape.md` 和 `comparables.md`（可比公司表）
- **硬约束**：所有数据点必须标注来源；找不到来源就标 `[未核实]`

### 环节 D：`/dd-team` `/dd-product` `/dd-market` `/dd-finance` 模块化尽调
- **拆成四个 skill**：避免单次对话上下文过载，每个模块独立产出
- **共同要求**：
  - 每个判断附 1 条以上证据
  - 每个判断附 1 条以上"如果这个判断错了会是什么样"
  - 输出末尾必须有"本次尽调没有覆盖的关键问题"清单

### 环节 E：`/founder-sync` 创始人访谈
- **访谈前**：根据当前尽调缺口生成定制化问题清单（不用通用模板）
- **访谈中**：录音
- **访谈后链路**（标准三步）：
  1. `/asr` 把录音转写成文本
  2. `/meeting-notes` 把转写 + 你的现场笔记整理成 6 段式 .docx
  3. Claude 在 .docx 基础上补一层分析：标记"创始人回避 / 含糊 / 主动澄清"的部分，写入 `03-founder-sync.md`
- **关键约定**：`/meeting-notes` 产出的 .docx 是事实层（原话+笔记），`03-founder-sync.md` 是判断层（解读+疑点），两者分开存放

### 环节 F：`/bear-case` 反向论证
- **独立环节，不和 memo 合并**：避免论证偏向化
- **动作**：扮演挑剔 LP / 竞品 CEO / 怀疑论同行三种角色，写出最锋利的"为什么不该投"
- **产出**：`04-bear-case.md`，至少 3 条不同维度的杀手论点

### 环节 G：`/ref-check` 背调整理
- **输入**：背调对话转写或要点（电话/语音 → `/asr` 转写 → 可选用 `/meeting-notes` 结构化）
- **动作**：聚合多源观点，标注共识 / 分歧 / 单点声明
- **硬约束**：单一信源的负面评价不能作为否定理由，必须显式标"待交叉验证"

### 环节 H：`/ic-memo` 投决备忘录
- **输入**：所有前序产出
- **结构**：交易摘要 → 公司 → 市场 → 产品 → 团队 → 财务 → 投资逻辑 → 风险与反方观点 → 估值与条款 → 决议建议
- **硬约束**：
  - "风险与反方观点"章节字数不少于"投资逻辑"章节
  - 所有数字必须可追溯到 `00-source/` 或外部链接
  - 不允许出现没有证据的形容词（"领先的"、"颠覆性的"等需要附数据或删除）

### 环节 I：`/decision` 归档
- **输入**：投决会结果
- **动作**：归档决议、估值、条款、关键分歧点
- **埋点**：写下"6 个月 / 12 个月后回看时要验证的 3 个假设"，供 `/retro` 使用

### 环节 J：`/retro` 复盘（跨项目）
- 周期性运行，对照 `99-decision.md` 中埋下的假设回看实际进展
- 目的：识别自己的判断偏差，而不是事后给项目打分

## 4. 跨环节共用资产

放在 `library/` 下：

- `memo-template.md` — IC memo 骨架
- `dd-checklist.md` — 各模块尽调要点清单
- `founder-questions-bank.md` — 按主题分类的提问库（用于 `/founder-sync` 生成定制问题）
- `comparables-db.md` — 长期沉淀的可比公司估值数据（手工维护，标注更新日期）
- `red-flags.md` — 历史上踩过的雷点清单（自己的，会随时间增长）

## 5. 运营辅助 skills（不在投资决策流程内）

下列 skill 不参与项目判断，但属于投资经理日常工作的一部分，单独运行：

### `/invoice-manager` — 差旅报销整理
- 触发时机：出差结束 / 月底报销
- 标准用法：把 PDF 发票放进 `input/` → `process_invoices.py` 做 OCR → 复核 JSON → `generate_summary.py --month YYYY-MM` → 得到 Excel 汇总 + 自动归档的 PDF
- 与投资流程的关系：**完全解耦**，但归档目录里的"项目"字段可以引用 `projects/{slug}/` 的命名，保持一致性
