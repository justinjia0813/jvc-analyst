## 1. 挑剔 LP 视角

| 字段 | 内容 |
| --- | --- |
| 核心不投理由 | 早期收入线索可能不足以支撑该轮估值和基金回报要求。 |
| 事实依据 | 创始人称公司年化收入 420 万元、覆盖 18 条产线，但未提供合同、回款和收入拆分。[创始人自述][未核实] |
| 推理链 | 如果收入主要来自一次性交付或低毛利硬件集成，那么当前收入不能证明高质量软件化增长，也难以支撑高倍数融资。[推测] |
| 最怕的验证问题 | 过去 12 个月实际确认收入、回款、毛利和续费分别是多少？ |
| 可证伪性 | 若公司能提供合同台账、银行回款、同客户多产线扩张和 60% 以上软件/服务毛利，这条担忧会被明显削弱。 |
| 下一步取证 | 要求公司提供收入 cohort、合同样本、回款截图、客户扩产线时间线。 |
| 来源标签 | S1, S2 |

## 2. 竞品 CEO 视角

| 字段 | 内容 |
| --- | --- |
| 核心不投理由 | 传统视觉集成商可能用价格和客户关系压制其标准化方案。 |
| 事实依据 | deck 提到目标客户已有人工质检和传统视觉方案，但没有披露替换成本、竞品报价或丢单原因。[示例素材: deck p.4][未核实] |
| 推理链 | 如果客户采购仍由设备商、系统集成商或老供应商主导，新公司的低代码平台不一定能绕开既有客户关系。[推测] |
| 最怕的验证问题 | 客户为什么愿意换掉已有供应商，且愿意承担导入风险？ |
| 可证伪性 | 若客户访谈显示采购决策来自明确 ROI、传统方案无法覆盖缺陷类型，且竞品报价明显高于本方案，这条担忧会被削弱。 |
| 下一步取证 | 访谈 3 个付费客户、2 个流失客户和 2 个传统视觉竞品渠道商。 |
| 来源标签 | S1, S3 |

## 3. 怀疑论同行视角

| 字段 | 内容 |
| --- | --- |
| 核心不投理由 | 工业 AI 项目容易从产品公司退化成交付项目制公司。 |
| 事实依据 | 产品宣称“低代码训练 + 边缘部署”，但未披露每个客户平均部署人天、模型复用率、售后支持成本。[示例素材: deck p.5-p.6][未核实] |
| 推理链 | 工业场景往往高度碎片化；如果每条产线都要重新采样、标注、调参和驻场，规模化销售会被交付成本吃掉。[推测] |
| 最怕的验证问题 | 新客户上线从样本采集到稳定验收平均需要多少人天？ |
| 可证伪性 | 若项目数据显示 70% 以上缺陷模型可复用、平均部署低于 5 人天、售后工单随客户数增长没有线性上升，这条担忧会被削弱。 |
| 下一步取证 | 查看最近 10 个项目的实施日志、工单记录、毛利拆分和交付复盘。 |
| 来源标签 | S1 |

## 交叉风险清单

| 风险 | 影响的投资假设 | 当前证据 | 缺口 | 下一步验证 |
| --- | --- | --- | --- | --- |
| 收入质量不清 | 商业化进展和估值支撑 | 年化收入为创始人自述 | 合同、回款、毛利、续费 | 财务底稿和客户访谈 |
| 竞品替代压力 | 技术节点和市场进入 | deck 只展示自身流程，单个客户提到传统供应商价格压力。[待交叉验证] | 竞品报价、客户替换动机 | 竞品访谈和采购链访谈 |
| 交付成本过高 | 标准化产品假设 | 未披露部署人天 | 项目实施数据 | 抽查项目日志 |

## 可证伪验证计划

| 优先级 | 待验证问题 | 推荐对象 | 需要看到的证据 | 若验证失败的含义 |
| --- | --- | --- | --- | --- |
| P0 | 收入是否可持续且毛利结构健康？ | 创始人、财务负责人、客户 | 合同、回款、收入拆分、毛利 | 商业化进展亮点需要下修 |
| P0 | 部署是否真的标准化？ | 交付负责人、客户现场负责人 | 实施日志、上线周期、工单数据 | 产品化假设需要下修 |
| P1 | 客户为何选择它而非传统集成商？ | 付费客户、流失客户、竞品渠道 | 采购记录、报价单、丢单原因 | 竞争优势需要下修 |
