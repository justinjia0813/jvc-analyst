# /bear-case 反向论证

`/bear-case` 用于把项目素材中的反面信号压成最锋利、可追溯、可证伪的不投理由。它是 `/bull-case` 的反向镜像，任务是系统找茬，不做平衡判断，也不替用户做最终投资决策。

## 输入

- deck、BP、官网、融资新闻、财务预测、交易条款
- `/prescreen` 初筛纪要、尽调笔记、客户访谈、专家访谈、背调记录
- `/bull-case` 投资亮点、`/track-research` 赛道研究、`/comps-dd` 竞品数据
- 用户当前最犹豫的点、直觉性担忧或投委会可能追问的问题

输入不足时不要补故事，用 `[需要用户提供]` 或 `[未核实]` 标出来。对创始人或项目的不利判断如果只有单点信源，必须标 `[待交叉验证]`。

## 三个反方角色

1. **挑剔 LP**：这个案子的回报/风险比、基金组合适配度、退出路径和资本效率是否说得通。
2. **竞品 CEO**：如果我是竞争对手，如何在产品、渠道、价格、客户关系或技术路线中赢它。
3. **怀疑论同行**：历史上相似故事为什么失败；团队、市场、时机、商业模式是否有结构性硬伤。

## 输出要求

- 至少输出 3 条不同维度的反对论点，不能只围绕同一个风险换说法。
- 每条论点都必须包含：
  - `核心不投理由`：一句话写清为什么这个点足以影响继续投入时间或资金。
  - `事实依据`：来自素材、访谈或公开来源的可追溯事实；不足则标 `[未核实]`。
  - `推理链`：事实如何推导到风险，不跳步，不用情绪化判断。
  - `可证伪性`：什么证据出现后，这条担忧可以被明显削弱或推翻。
  - `下一步取证`：可直接向创始人、客户、前员工、竞品或专家验证的问题。

## 输出格式

使用 `templates/bear-case-template.md` 作为固定结构，输出 Markdown。

## 质量红线

- 不写“但另一方面”“当然也有机会”等平衡段落；本环节只负责反向论证。
- 不写“建议不投”“pass 掉”等终局判断，最多写“该风险若成立，会削弱某条投资假设”。
- 不把情绪、偏见或单点传闻写成事实；必须保留 `[用户直觉]`、`[待交叉验证]`、`[未核实]` 标签。
- 不为了显得尖锐而扩大事实。负面论点的杀伤力来自证据和推理，不来自措辞。
- 产出应能直接进入 `/ic-memo` 的 `风险与反方观点` 章节，并保留来源标签。

## 反馈迭代空间

真实项目跑完后，优先根据以下反馈调整模板：

- 哪类反方论点最能改变后续尽调方向
- 投委会最常追问的是 LP 视角、竞品视角还是历史类比
- `可证伪性` 是否足够具体，能不能转化为可执行问题
- 单点负面信息的标签是否足够保护结论边界
