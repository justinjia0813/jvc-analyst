# /bull-case 投资亮点

`/bull-case` 用于把项目素材中的正向信号压缩成最锋利、可追溯、可讨论的投资亮点。它是 `/bear-case` 的正向镜像，但不替用户做投资决策。

## 输入

- deck、BP、官网、融资新闻、访谈纪要
- `/prescreen` 初筛纪要、尽调笔记、客户访谈、专家访谈
- 用户已经感觉“值得认真看”的直觉或关键词
- 可选：已有反方观点，用于把亮点写得更能回应质疑

输入不足时不要补故事，用 `[需要用户提供]` 或 `[未核实]` 标出来。

## 四个固定层面

1. **行业趋势**：赛道是否进入拐点，需求、政策、成本、供给侧是否发生变化。
2. **技术节点**：技术是否到了可用、可部署、成本可接受或性能跃迁的阶段。
3. **团队优势**：团队是否有独特认知、资源、履历、交付经验或客户入口。
4. **商业化进展**：客户、收入、复购、回款、渠道、毛利、交付效率是否出现早期验证。

## 输出层级

- **标题级**：一句话亮点，适合放进 memo 或投委会提纲。
- **正文级**：每条亮点必须包含：
  - `事实依据`：来自素材的可追溯事实，不足则标 `[未核实]`
  - `判断论点`：为什么这些事实构成投资亮点
  - `仍需验证`：这个亮点最怕被哪条证据推翻

## 输出格式

使用 `templates/bull-case-template.md` 作为固定结构，输出 Markdown。

## 质量红线

- 不写“建议投资”“必投”“强烈推荐”等终局判断。
- 不把愿景当事实；愿景、规划、口头承诺必须标 `[创始人自述]`。
- 每个亮点至少有一个事实依据；没有事实时只能写入“待验证亮点”。
- 不为了正向论证隐藏风险；每条亮点都要写 `仍需验证`。
- 正文级论证要能被复制到 `/ic-memo` 的投资逻辑部分，但仍保留来源标签。

## 反馈迭代空间

真实项目跑完后，优先根据以下反馈调整模板：

- 哪类亮点最容易被投委会追问
- 哪些事实标签不够细，导致后续查证困难
- 标题级亮点是否过于抽象，无法直接放进 memo
- 正文级论证是否能自然迁移到 `/ic-memo`
